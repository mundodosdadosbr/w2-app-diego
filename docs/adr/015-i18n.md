# ADR-015: i18n — UI em pt-BR, conteúdo de estudo em EN com tradução de apoio

- **Status**: Accepted — **Amended 2026-04-21**: conteúdo de estudo **apenas en-US** no MVP
- **Data**: 2026-04-21
- **Tags**: i18n, pedagogia, frontend

> **Nota de atualização (2026-04-21):** o conteúdo de estudo é **exclusivamente en-US** no MVP (não mais "en-US primário, en-GB desejável"). Justificativa pedagógica: brasileiros consomem predominantemente mídia americana (streaming, música, tech, acadêmico, negócios); en-US tem fonologia mais acessível para falantes de pt-BR. Implicações:
>
> - TTS: vozes en-US no ElevenLabs (candidatos: Rachel, Adam, Matilda — selecionar).
> - STT: Amazon Transcribe em `en-US` obrigatório ([ADR-016](016-stt-amazon-transcribe.md)).
> - Spelling: americano (color, center, organize, analyze) em todo conteúdo seed.
> - Prompts das personas IA ([knowledge/10](../knowledge/10-ai-personas-e-prompts.md)) instruem: julgar contra American English; aceitar formas britânicas mas sugerir preferencialmente a americana.
> - en-GB fica para reavaliação pós-PMF se houver demanda.

## Contexto

Temos duas "línguas" coexistindo:

1. **UI da plataforma** — botões, menus, mensagens de erro, labels, feedback: **pt-BR**. Público-alvo é falante de português brasileiro.
2. **Conteúdo de estudo** — frases, diálogos, vocabulário: **inglês**. É o material de aprendizagem.

Além disso, o aluno precisa de **tradução de apoio** (pt-BR ao lado ou sob demanda) para vocabulário e frases, sem que isso vire "muleta" permanente.

Decisões a tomar:
- Onde e como separar UI strings vs conteúdo do curso.
- Como modelar tradução do conteúdo.
- Estratégia para futuros idiomas de UI (espanhol, por exemplo) no médio prazo.

## Decisão

### UI strings (pt-BR)
- **`next-intl`** para i18n do Next App Router.
- Todas as strings da UI em `messages/pt-BR.json`.
- Um só idioma no MVP; arquitetura já suporta adicionar `en.json`, `es.json`.
- Chaves organizadas por feature: `dashboard.*`, `lesson.player.*`, `auth.*`.

### Conteúdo do curso (EN com tradução pt-BR)
- Modelado no banco com campos `en` e `pt_br` **no próprio registro**:
  - `vocabulary_items { en, pt_br, audio_key, part_of_speech, example_en, example_pt_br }`
  - `phrase_patterns { en, pt_br, audio_key, function, context }`
  - `dialog_lines { en, pt_br, speaker, order, audio_key }`
  - `grammar_points { title_pt_br, explanation_pt_br, rule_pattern_en, examples[] }`
- **`pt_br` é opcional** em alguns casos (ex.: exemplos extras gerados na hora pelo tutor IA). Quando ausente, a UI simplesmente não oferece tradução.

### Princípios pedagógicos de tradução
- Tradução **não aparece por padrão** em frases e diálogos — é **toggle** controlado pelo aluno.
- Em vocabulário novo (bloco New Words), tradução aparece ao lado por padrão — o aluno está no momento de INPUT.
- Em drills e OUTPUT, tradução fica **escondida por padrão**. Aluno clica para revelar.
- Explicações gramaticais (grammar points) em pt-BR no nível iniciante, com transição gradual para EN nos níveis mais altos (estratégia: campo `explanation_en` passa a ser preferido quando `user.level >= B1`).

### Detalhes técnicos
- **Rotas não prefixadas** no MVP (`/dashboard` em vez de `/pt/dashboard`), já que só há um idioma. Quando entrar um segundo, migramos para `/[locale]/...`.
- **Datas, números, moeda** via `Intl.*` com `pt-BR`.
- **Nomes de dias/meses** em strings de UI; no conteúdo de curso, em inglês.
- **Formulários**: mensagens de validação Zod traduzidas via `zod-i18n-map`.

### Nível do aluno influencia UI?
No MVP, não. A UI é constante pt-BR. O que varia por nível é o **conteúdo** (vocab permitido, velocidade do TTS, voz do tutor IA), não a UI.

## Alternativas consideradas

- **`react-intl` / `formatjs`** — maduro, mas API verbosa para App Router. `next-intl` é mais idiomático em Server Components.
- **Content duplicado em tabelas separadas (`vocab_items_en`, `vocab_items_pt`)** — overhead inútil: cada "item lexical" é um par EN↔pt-BR; mantê-los juntos reflete o domínio.
- **Tradução automática por LLM em runtime** — risco de inconsistência e custo. Tradução humana no conteúdo publicado + LLM apenas em casos dinâmicos (explicação do tutor quando aluno pergunta).
- **UI em EN** — contraria o público-alvo brasileiro iniciante; barreira de entrada alta.

## Consequências

### Positivas
- UI clara e direta para brasileiros.
- Tradução como apoio, não como muleta — alinhada à pedagogia de imersão gradual.
- Estrutura de dados simples: um par EN/pt-BR por item.
- Caminho claro para adicionar outros idiomas de UI depois.

### Negativas / Custos aceitos
- Campos `pt_br` inflam tabelas — overhead pequeno comparado ao valor pedagógico.
- Aluno avançado pode achar UI em pt-BR infantilizada. Mitigação: toggle "EN UI" opcional em configurações (backlog).
